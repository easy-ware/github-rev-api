The gh-manager API server provides a RESTful interface for interacting with GitHub repositories. It allows developers to programmatically manage repositories, files within repositories, releases, and release assets without directly using the GitHub web interface or the git command line for these specific tasks.

This API acts as a wrapper around the PyGithub library, exposing common operations via HTTP endpoints secured by GitHub Personal Access Tokens (PATs). It features structured JSON responses, robust error handling, and a dedicated workflow for handling release asset uploads.

info

Purpose: This document details how to set up, run, and interact with the API server, covering prerequisites, core concepts, and detailed endpoint specifications.

Scope

This documentation covers Version 1.0 of the API server, based on the provided server.py script. Future versions may introduce changes or additional features not covered here.

Prerequisites

  • Python 3: Ensure you have Python 3.8 or newer installed. You can check with python --version or python3 --version.
  • pip: Python's package installer, usually included with Python. Verify with pip --version or pip3 --version.
  • GitHub Personal Access Token (PAT): You need a PAT from GitHub with appropriate permissions (scopes) for the operations you intend to perform.
    • Minimum required: repo (Full control of private repositories)
    • For full functionality (including deleting test repos): repo, delete_repo
    link

    Generate a new PAT in your GitHub settings: github.com/settings/tokens. Remember to copy the token immediately as it won't be shown again.

Installation

  1. Save the Server Code: Obtain the Python server script and save it as server.py in a dedicated directory.
  2. Install Dependencies: Open your terminal or command prompt, navigate (`cd`) into the directory where you saved server.py, and run: 
    pip install Flask PyGithub python-dotenv requests

    This command installs Flask (the web framework), PyGithub (the GitHub API library), python-dotenv (for environment variables), and requests (used internally and by the helper script).

Running the Server

Environment Variables (Optional)

Configure the server's host, port, and debug mode using environment variables for flexibility:

  • HOST: IP address to bind to. Default: 127.0.0.1 (local access). Use 0.0.0.0 for network access (be careful).
  • PORT: Port number. Default: 5000.
  • FLASK_DEBUG: Set to 1 or true for debug mode (auto-reload, better errors). Default: false.
warning

Security Risk: Never run the server with FLASK_DEBUG=1 in a production environment. It can expose sensitive information and allow arbitrary code execution.

You can set these directly in your shell:

# Example for Linux/macOS
export FLASK_DEBUG=1
export PORT=5001

Alternatively, create a file named .env in the same directory as server.py:

# .env file content
FLASK_DEBUG=true
HOST=0.0.0.0
PORT=5001
# GITHUB_TOKEN=... # Not typically needed here, passed via request headers

The server uses python-dotenv to automatically load variables from this .env file if it exists.

Starting the Server

Choose one of the following methods:

  1. Using the flask command (Recommended): 
    # Ensure FLASK_APP is set (or name your file app.py)
export FLASK_APP=server.py
# Optionally set FLASK_DEBUG, HOST, PORT if not using .env

# Run the server (adjust host/port if needed)
flask run --host=0.0.0.0 --port=5000
  2. Directly using python: 
    # Environment variables (shell or .env) will be used for config
python server.py

If successful, you should see output indicating the server is running, usually including the address like `http://127.0.0.1:5000/`.

Authentication

All functional API endpoints (excluding the root / and helper script endpoint) require authentication using a GitHub Personal Access Token (PAT). The server expects the token in **every request** via specific HTTP headers.

Provide the token using one of these headers:

  1. X-API-Token: YOUR_GITHUB_PAT
  2. Authorization: Bearer YOUR_GITHUB_PAT

Replace YOUR_GITHUB_PAT with your actual token.

security

Security Best Practice: Treat your PAT like a password. Do not embed it directly in client-side code or commit it to version control. Use environment variables or secure configuration methods to manage tokens.

Token Scopes

The required permissions (scopes) for your PAT depend on the actions you perform:

  • Essential: repo scope is needed for almost all operations (creating repos, managing files, releases, assets in private *and* public repos).
  • Optional: delete_repo scope is required *only* if you intend to use this API to delete repositories (which is not currently an endpoint in the provided server code, but good to know for potential extensions).

If a request fails with a 401 Unauthorized or 403 Forbidden error, double-check your token and its associated scopes in your GitHub settings.

Base URL

The Base URL is the root address where the API server is accessible. By default, this is http://127.0.0.1:5000 when running locally.

All endpoint paths documented later are relative to this Base URL. For example, the "Create Repository" endpoint path is /repo/create, so the full URL would be http://127.0.0.1:5000/repo/create.

Request Format

  • Method: Use the correct HTTP method (GET, POST, PUT, DELETE) specified for each endpoint.
  • Headers:
    • Include the authentication header (X-API-Token or Authorization: Bearer).
    • For requests with a body (POST, PUT), set the Content-Type header to application/json.
  • URL Parameters: Some endpoints include variables in the path itself (e.g., /repo/{owner}/{repo_name}/file). Replace placeholders like {owner} with actual values.
  • Request Body: For POST and PUT requests requiring data, send a JSON object in the request body as specified in the endpoint documentation.

Response Format

The API uses JSON for all responses (except potentially the raw script download).

Success Responses

  • Status Code: Typically 200 OK (successful GET/PUT/DELETE), 201 Created (successful POST creating a resource), or 204 No Content (success, no body needed).
  • Body Structure: 
    {
  "success": true,
  "data": {
    // Results specific to the endpoint appear here
    "message": "Optional success message (e.g., Repository created)",
    "status": "Optional status indicator (e.g., uploaded, deleted)",
    // ... other data fields like id, url, etc.
  }
}

Error Responses

  • Status Code: Standard HTTP error codes (4xx for client errors, 5xx for server errors). See Error Handling below.
  • Body Structure: 
    {
  "success": false,
  "error": {
    "code": "ERROR_CODE_STRING", // A machine-readable error identifier
    "message": "A human-readable explanation of the error.",
    "details": "..." // Optional: More specific info, often from GitHub's API
  }
}

Error Handling

The API provides feedback through HTTP status codes and detailed JSON error bodies.

Common HTTP Status Codes

  • 200 OK: Success.
  • 201 Created: Resource created.
  • 204 No Content: Success, no body.
  • 400 Bad Request: Invalid request (missing data, bad format).
  • 401 Unauthorized: Authentication required or failed.
  • 403 Forbidden: Authenticated, but lacking permissions (check PAT scopes).
  • 404 Not Found: Resource (repo, file, release, etc.) or API route not found.
  • 405 Method Not Allowed: Incorrect HTTP verb used for the endpoint.
  • 409 Conflict: Action conflicts with current state (e.g., asset name exists).
  • 422 Unprocessable Entity: GitHub validation failed (e.g., invalid repo name, tag missing).
  • 500 Internal Server Error: Unexpected server-side issue.
  • 502 Bad Gateway: Network error contacting GitHub.
  • 504 Gateway Timeout: GitHub request timed out.

Custom Error Codes (in JSON error.code)

These provide more specific reasons within the JSON error body:

  • AUTH_REQUIRED
  • AUTH_FAILED
  • MISSING_PARAMETER
  • INVALID_INPUT
  • BAD_REQUEST
  • RESOURCE_NOT_FOUND
  • ROUTE_NOT_FOUND
  • METHOD_NOT_ALLOWED
  • VALIDATION_ERROR
  • GITHUB_API_ERROR (Check details)
  • RESOURCE_CONFLICT
  • NETWORK_ERROR
  • INTERNAL_ERROR

Always check the error object in the JSON response for detailed diagnostics when a request fails.

Uploading release assets to GitHub via this API involves a specific two-step workflow designed to avoid the API server needing direct access to your local files.

sync_alt

Workflow Overview: Prepare on API Get Details Run Client Script Upload to GitHub.

Step-by-Step Process

  1. Step 1: Prepare Upload (Client API)

    Send a POST request to the Prepare Asset Upload endpoint: /repo/{owner}/{repo}/release/{id_or_tag}/asset.

    Include the local path to your asset file (asset_path) and optionally the desired asset_name in the JSON body. Provide your PAT via headers.

    See the Prepare Asset Upload Request Examples for `curl`, Python, and Postman samples of this step.

  2. Step 2: Receive Validation & Details (API Client)

    The API server validates the request (repo/release existence, token permissions, potential name conflicts). It **does not** access the local asset path provided.

    If valid, the API responds with 200 OK and a JSON body containing the crucial upload_details object. This object includes the specific GitHub URL needed for the direct upload (release_upload_url).

    {
  "success": true,
  "data": {
    "message": "Asset validated...",
    "upload_details": {
      "repo_full_name": "...",
      "release_id_or_tag": "...",
      "asset_path_local": "/local/path/to/my_asset.zip", // For client reference
      "asset_name_remote": "asset-v1.zip",
      "release_upload_url": "https://uploads.github.com/..." // Crucial URL
    }
  }
}

    Important: Save this entire upload_details JSON object (or pass it directly) for the next step.

  3. Step 3: Obtain Helper Script (Client)

    If you don't have it already, download the upload.py script from the API server:

    curl http://YOUR_API_SERVER/get/v1/upload-script -o upload.py
chmod +x upload.py # Make executable if needed

    See the Get Upload Script Request Examples for how to do this programmatically.

  4. Step 4: Execute Upload Script (Client)

    Run the downloaded upload.py script on your local machine where the asset file resides. Provide two arguments:

    1. The path to a file containing the JSON upload_details from Step 2, OR the JSON string itself.
    2. Your GitHub PAT using the --token argument (or set the `GITHUB_PAT` environment variable).
    # Option A: Save details to a file (e.g., details.json from Step 2)
# Assuming upload_details.json contains the JSON output from Step 2
python upload.py upload_details.json --token YOUR_GITHUB_PAT

# Option B: Pass JSON as a string (ensure proper shell quoting!)
# (Less practical for complex JSON, prefer file method)
# UPLOAD_DETAILS_JSON='{"repo_full_name": ..., "release_upload_url": ...}' # Populate this
# python upload.py "$UPLOAD_DETAILS_JSON" --token YOUR_GITHUB_PAT

# Option C: Using Environment Variable for Token
# export GITHUB_PAT="YOUR_GITHUB_PAT"
# python upload.py upload_details.json
  5. Step 5: Actual Upload (Helper Script GitHub)

    The upload.py script performs the following:

    • Reads the upload_details JSON.
    • Uses the provided --token (or environment variable) to authenticate with GitHub.
    • Opens and reads the local file specified by asset_path_local from the details JSON.
    • Uploads the file content directly to the specific release_upload_url provided by the API in Step 2, using the asset_name_remote.
    • Prints progress and success/failure messages to the console.

Rationale for this Workflow

This multi-step process ensures:

  • Security: The central API server never needs access to your local filesystem or the potentially large asset files themselves.
  • Efficiency: The heavy lifting of file transfer happens directly between the client machine (where the file is) and GitHub's upload infrastructure.
  • Flexibility: The API handles validation and coordination, while the client script handles the OS-specific file access and upload mechanics.

  • Request/Response: Use JSON for request bodies and expect JSON responses. Pay attention to HTTP status codes and the error object for diagnostics.
  • Endpoints: Refer to the specific API Endpoints documentation for details on paths, parameters, and expected behavior. Use the provided `curl`, Python, and Postman examples as starting points.
  • Asset Uploads: Follow the dedicated two-step workflow involving the "Prepare Asset Upload" endpoint and the client-side upload.py script.
